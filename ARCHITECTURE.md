# Pi Helper Bot — Architecture & Documentation

## Overview

A self-programming Telegram bot that runs 24/7 on a Raspberry Pi. You talk to it in
natural language, and it creates **skills** — recurring scheduled messages or custom
commands — that run locally without needing an LLM at execution time.

**Core principle: LLM at creation time, pure local execution at runtime.**

---

## System Requirements

- Raspberry Pi 4/5 (2GB+ RAM) running Raspberry Pi OS (64-bit)
- Python 3.11+
- A Telegram Bot Token (from @BotFather)
- A Claude API key (Haiku is sufficient and cheap)
- A private Git remote (GitHub, Gitea, etc.) for state backup
- Internet connection (for Telegram API, weather/sports APIs, and LLM calls during skill creation)

---

## Key Design Decisions

### 1. Skills are data, not code (Tier 1)

~80% of use cases are handled by **template skills**: JSON files that define a trigger,
a data source (HTTP API), and a Jinja2 message template. The skill executor is a
deterministic engine that reads JSON → fetches data → renders template → sends message.
No LLM is involved at runtime.

Example skill JSON (`skills/morning_weather.json`):
```json
{
  "id": "morning_weather",
  "name": "Morning Weather",
  "description": "Daily weather report for workdays",
  "enabled": true,
  "created_at": "2026-03-14T08:00:00Z",
  "trigger": {
    "type": "cron",
    "cron": "0 7 * * 0-4",
    "timezone": "Asia/Jerusalem"
  },
  "data_sources": [
    {
      "id": "weather",
      "type": "http",
      "url": "https://api.open-meteo.com/v1/forecast",
      "params": {
        "latitude": 32.08,
        "longitude": 34.78,
        "current": "temperature_2m,weather_code,precipitation,wind_speed_10m",
        "daily": "precipitation_probability_max,temperature_2m_max,temperature_2m_min",
        "timezone": "Asia/Jerusalem",
        "forecast_days": 1
      },
      "method": "GET",
      "timeout": 15
    }
  ],
  "message_template": "🌤 *Morning Weather — {{ now.strftime('%A, %b %d') }}*\n\n🌡 Currently: {{ weather.current.temperature_2m }}°C\n🔼 High: {{ weather.daily.temperature_2m_max[0] }}°C | 🔽 Low: {{ weather.daily.temperature_2m_min[0] }}°C\n🌧 Rain chance: {{ weather.daily.precipitation_probability_max[0] }}%\n💨 Wind: {{ weather.current.wind_speed_10m }} km/h\n\n{% if weather.daily.precipitation_probability_max[0] > 50 %}☂️ *Bring an umbrella!*{% elif weather.daily.precipitation_probability_max[0] > 25 %}🤔 Might want an umbrella just in case{% else %}😎 No rain expected{% endif %}",
  "parse_mode": "Markdown"
}
```

### 2. Complex skills are Python plugins (Tier 2)

For things that can't be expressed as fetch+template (scraping, multi-step logic,
data transformation), the system supports Python plugin files in `plugins/`.
Each plugin exports a single async function:

```python
async def execute(context: PluginContext) -> str:
    """Returns the formatted message string to send."""
    ...
```

These are generated by Claude Code or written manually, reviewed by the user,
and committed to git.

### 3. Git-backed state (no database as source of truth)

All state is stored as flat files in a git repository:
- `skills/*.json` — template skill definitions
- `plugins/*.py` — complex skill code
- `memory.md` — persistent user preferences and context
- `changelog.md` — auto-generated log of all changes

On every mutation (skill create/edit/delete, memory update), the bot:
1. Writes the file
2. Commits with a descriptive message
3. Pushes to the remote

**Recovery**: clone the repo on a new Pi, set env vars, start the service.

SQLite is used ONLY as a volatile runtime cache (API response caching, execution
logs). It is NOT the source of truth and can be deleted without data loss.

### 4. Memory system

`memory.md` stores persistent context that gets injected into every LLM API call.
Categories:
- **Preferences**: tone, language, units, formatting
- **Facts**: location, work schedule, interests, allergies
- **Bot behavior**: confirmation requirements, default settings

When the user says something like "from now on keep answers short," the LLM
classifies this as a memory update (not a skill creation), updates `memory.md`,
and confirms.

### 5. Message classification

Every incoming message is classified into one of three intents:
1. **create_skill** — "Send me weather every morning" → create/edit a skill
2. **update_memory** — "Remember that I'm vegetarian" → update memory.md
3. **chat** — "What time is it?" → respond directly (one-off LLM call)

Classification is done by the LLM with a structured output schema.

## Component Details

### bot/main.py
- Creates the `Application` (python-telegram-bot v21+)
- Loads all skills from `skills/` and registers their triggers
- Loads all plugins from `plugins/` and registers their triggers
- Sets up the ConversationHandler for skill creation
- Sets up command handlers (/start, /skills, /enable, /disable, /delete)
- Starts polling (NOT webhooks — simpler for Pi, no SSL needed)
- Handles graceful shutdown (SIGTERM from systemd)

### bot/config.py
- `TELEGRAM_TOKEN` — from env
- `CLAUDE_API_KEY` — from env
- `ALLOWED_USER_IDS` — list of Telegram user IDs that can interact (security)
- `GIT_REMOTE` — remote URL for git push
- `GIT_BRANCH` — default "main"
- `TIMEZONE` — default from env, e.g. "Asia/Jerusalem"
- `LOCATION_LAT`, `LOCATION_LON` — default location for weather etc.
- `DOMAIN_WHITELIST` — allowed domains for HTTP fetches in skills
- `MAX_SKILLS` — limit per user (default 50)
- `SKILL_EXECUTION_TIMEOUT` — seconds (default 30)

### bot/core/skill_registry.py
- `load_all_skills()` → reads all `skills/*.json`, validates with Pydantic
- `save_skill(skill)` → writes JSON, calls git_sync
- `delete_skill(skill_id)` → removes file, calls git_sync
- `get_skill(skill_id)` → returns single skill
- `list_skills()` → returns all skills with status

### bot/core/skill_executor.py
- `execute_skill(skill)` → the main runtime loop:
  1. For each data_source: call `data_fetcher.fetch()`
  2. Build template context: `{source_id: response_data, now: datetime}`
  3. Render `message_template` through Jinja2 SandboxedEnvironment
  4. Send via Telegram with parse_mode
- Custom Jinja2 filters: `weather_description` (code→emoji), `format_date`, `number_format`
- Error handling: retry once, then disable skill + notify user after 3 consecutive failures

### bot/core/skill_scheduler.py
- Wraps python-telegram-bot's JobQueue (which wraps APScheduler)
- `register_skill(skill)` → creates a cron or command trigger
- `unregister_skill(skill_id)` → removes the job
- `reload_all()` → unregister all, re-register from registry

### bot/core/plugin_loader.py
- `load_plugin(name)` → `importlib.import_module(f"plugins.{name}")`
- Validates that the module has `execute(context) -> str`
- Validates that the module has `TRIGGER` dict (same format as skill trigger)
- Wraps execution with timeout and error catching

### bot/core/data_fetcher.py
- Async httpx client with connection pooling
- Domain whitelist enforcement (rejects URLs not on the list)
- Response caching via `cache.py` with configurable TTL per domain
- Default TTLs: weather=1800s, sports=21600s, news=3600s
- Timeout: 15 seconds per request

### bot/core/git_sync.py
- `commit_and_push(filepath, message)` → git add, commit, push
- Runs git commands via `asyncio.create_subprocess_exec`
- Non-blocking: push failures are logged but don't block the bot
- Batches rapid changes (debounce 5 seconds)

### bot/core/cache.py
- SQLite with WAL journal mode
- `get(key)` → returns cached value if TTL not expired
- `set(key, value, ttl)` → stores with expiry timestamp
- `cleanup()` → periodic removal of expired entries
- Database lives in `data/cache.db` (gitignored)

### bot/handlers/message_router.py
- The main entry point for all non-command messages
- Sends message to LLM for classification
- Routes to: skill_creator, memory_handler, or chat_handler
- Security: checks `ALLOWED_USER_IDS` before processing

### bot/handlers/skill_creator.py
- ConversationHandler with states: AWAITING_DESCRIPTION, CONFIRMING, EDITING
- Flow:
  1. Receives natural language description
  2. Sends to LLM with skill-generation prompt + memory context
  3. Receives structured JSON skill definition
  4. Displays formatted preview to user with inline keyboard:
     ✅ Confirm | ✏️ Edit | ❌ Cancel
  5. On confirm: save_skill → register_scheduler → git_sync
  6. On edit: ask what to change → re-generate → back to preview

### bot/handlers/skill_manager.py
- `/skills` — list all skills with status (enabled/disabled/error)
- `/enable <id>` — enable a disabled skill
- `/disable <id>` — disable without deleting
- `/delete <id>` — delete with confirmation
- `/run <id>` — manually trigger a skill for testing
- `/edit <id>` — enter edit flow for existing skill

### bot/handlers/memory_handler.py
- Receives memory-update intent from router
- Reads current `memory.md`
- Sends to LLM: "Given current memory and this instruction, produce updated memory.md"
- Shows diff preview to user
- On confirm: write file → git_sync

### bot/handlers/chat_handler.py
- One-off LLM calls for general questions
- Includes memory.md content in system prompt
- Supports follow-up context within the same conversation

### bot/llm/client.py
- `async def classify(message) -> Intent` — message classification
- `async def generate_skill(description, memory) -> SkillDefinition` — skill creation
- `async def update_memory(current_memory, instruction) -> str` — memory update
- `async def chat(message, memory) -> str` — general chat
- Uses Claude Haiku (fast + cheap) for classification
- Uses Claude Sonnet for skill generation (better structured output)
- Retry with exponential backoff (3 attempts)
- Error handling: if API is down, inform user and suggest trying later

### bot/llm/prompts.py
- `CLASSIFICATION_PROMPT` — system prompt for intent classification
- `SKILL_GENERATION_PROMPT` — system prompt with all available data sources,
  trigger types, Jinja2 filter docs, and 3-5 example skills
- `MEMORY_UPDATE_PROMPT` — system prompt for memory editing
- `CHAT_PROMPT` — general assistant with memory context

### bot/llm/schemas.py
- Pydantic models that map to Claude's structured output:
  - `Intent` — enum: create_skill, update_memory, chat
  - `SkillDefinition` — full skill JSON schema
  - `DataSource` — HTTP source definition
  - `Trigger` — cron or command trigger
  - `MemoryUpdate` — updated memory text + summary of changes

## Data Flow Diagrams

### Skill creation flow
```
User: "Send me weather every morning at 7am on workdays"
  │
  ├─→ message_router.py: classify intent → "create_skill"
  │
  ├─→ skill_creator.py: send to LLM with SKILL_GENERATION_PROMPT + memory.md
  │     │
  │     └─→ Claude returns SkillDefinition JSON
  │
  ├─→ Show preview: "I'll create 'Morning Weather' — runs Mon-Fri 7:00 AM..."
  │     │
  │     └─→ User taps ✅ Confirm
  │
  ├─→ skill_registry.save_skill() → writes skills/morning_weather.json
  ├─→ skill_scheduler.register_skill() → APScheduler cron job
  └─→ git_sync.commit_and_push("Add skill: Morning Weather")
```

### Skill execution flow (runtime, no LLM)
```
APScheduler fires cron job at 7:00 AM
  │
  ├─→ skill_executor.execute_skill(morning_weather)
  │     │
  │     ├─→ data_fetcher.fetch("https://api.open-meteo.com/...") → JSON
  │     ├─→ Jinja2 render template with weather data
  │     └─→ telegram.send_message(chat_id, rendered_text)
  │
  └─→ Log execution result in cache.db
```

### Memory update flow
```
User: "Remember I prefer Celsius and I live in Tel Aviv"
  │
  ├─→ message_router.py: classify intent → "update_memory"
  │
  ├─→ memory_handler.py: read memory.md, send to LLM
  │     │
  │     └─→ Claude returns updated memory.md content
  │
  ├─→ Show diff: "I'll add: location=Tel Aviv, units=Celsius"
  │     │
  │     └─→ User taps ✅ Confirm
  │
  ├─→ Write memory.md
  └─→ git_sync.commit_and_push("Update memory: added location, unit preference")
```

## Available Data Sources (pre-configured in domain whitelist)

### Weather — Open-Meteo (no API key needed)
- Base URL: `https://api.open-meteo.com/v1/forecast`
- Params: latitude, longitude, current, daily, hourly, timezone
- Cache TTL: 30 minutes
- Docs: https://open-meteo.com/en/docs

### Football — football-data.org (free API key required)
- Base URL: `https://api.football-data.org/v4`
- Headers: `{"X-Auth-Token": "your-key"}`
- Key endpoints:
  - `/teams/{id}/matches?status=SCHEDULED&limit=1` — next match
  - `/competitions/PL/standings` — Premier League table
  - Manchester United team ID: 66
- Free tier: 10 requests/minute
- Cache TTL: 6 hours

### General HTTP
- Any domain on the whitelist can be fetched
- Default whitelist: api.open-meteo.com, api.football-data.org,
  wttr.in, api.exchangerate-api.com, newsapi.org
- Users can extend the whitelist via config

## Jinja2 Custom Filters

Available in all skill templates:
- `{{ code | weather_description }}` — WMO weather code → emoji + text
- `{{ date | format_date('%A, %b %d') }}` — date formatting
- `{{ num | round_num(1) }}` — round to N decimals
- `{{ text | truncate(100) }}` — truncate with ellipsis
- `{{ ts | from_timestamp }}` — unix timestamp → datetime
- `{{ dt | relative_time }}` — "in 3 hours", "2 days ago"

---

## Security Model

1. **User whitelist**: Only `ALLOWED_USER_IDS` can interact with the bot
2. **Domain whitelist**: Skills can only fetch from pre-approved domains
3. **No code execution in Tier 1**: Template skills can't run arbitrary code
4. **Jinja2 sandboxing**: SandboxedEnvironment prevents template injection
5. **Confirmation flow**: Every skill creation and memory update requires user approval
6. **Execution timeout**: 30 seconds per skill execution
7. **Rate limiting**: Max 5 skill creations per hour, max 50 total skills
8. **Plugin review**: Tier 2 plugins must be manually reviewed before activation

## Deployment

### Initial setup
```bash
# On the Pi
git clone <your-repo-url> ~/pi-helper-bot
cd ~/pi-helper-bot
python3 -m venv venv
source venv/bin/activate
pip install -r requirements.txt

# Configure
cp .env.example .env
nano .env  # Fill in TELEGRAM_TOKEN, CLAUDE_API_KEY, etc.

# Set up git for auto-push
git remote set-url origin git@github.com:you/pi-helper-bot.git

# Install and start systemd service
sudo cp mybot.service /etc/systemd/system/
sudo systemctl daemon-reload
sudo systemctl enable mybot
sudo systemctl start mybot

# Check logs
journalctl -u mybot -f
```

### systemd unit file (mybot.service)
```ini
[Unit]
Description=Pi Helper Telegram Bot
After=network-online.target
Wants=network-online.target

[Service]
Type=simple
User=pi
WorkingDirectory=/home/pi/pi-helper-bot
ExecStart=/home/pi/pi-helper-bot/venv/bin/python3 -m bot.main
Restart=always
RestartSec=10
KillSignal=SIGINT
StandardOutput=journal
StandardError=journal
Environment=PYTHONUNBUFFERED=1

[Install]
WantedBy=multi-user.target
```

### SD card longevity
- SQLite uses WAL journal mode (fewer writes)
- Cache DB lives in `data/` (can be on tmpfs if needed)
- Git pushes are debounced (5 second batching)
- Consider booting from USB SSD for long-term use

## Troubleshooting Guide

### Bot won't start
1. Check logs: `journalctl -u mybot -n 50`
2. Verify env vars: `cat .env` (token, api key present?)
3. Test manually: `cd ~/pi-helper-bot && source venv/bin/activate && python -m bot.main`
4. Check Python version: `python3 --version` (needs 3.11+)

### Skills not firing
1. Check skill is enabled: send `/skills` to the bot
2. Check timezone in skill JSON matches system timezone
3. Check cron expression: use https://crontab.guru to validate
4. Check APScheduler logs in journald
5. Manual test: `/run <skill_id>`

### API fetch failures
1. Check domain is in whitelist (config.py)
2. Test URL manually: `curl -s "the-url" | head`
3. Check if Pi has internet: `ping 8.8.8.8`
4. Check cache: stale cache might mask API changes

### Git push failures
1. Check SSH key: `ssh -T git@github.com`
2. Check remote: `git remote -v`
3. Push failures are non-blocking — bot continues working
4. Manual fix: `cd ~/pi-helper-bot && git push origin main`

### LLM API errors
1. Check API key validity
2. Check Anthropic status: https://status.anthropic.com
3. Bot will retry 3 times with backoff
4. Classification and skill creation degrade gracefully (inform user)
5. Runtime skill execution is NOT affected (no LLM dependency)

## Extending the System

### Adding a new data source
1. Add the domain to `DOMAIN_WHITELIST` in config.py
2. Add default cache TTL in data_fetcher.py
3. Document the API in this file under "Available Data Sources"
4. Skills can now reference the new source

### Adding a custom Jinja2 filter
1. Add the filter function in `skill_executor.py` → `_setup_jinja_env()`
2. Document it above under "Jinja2 Custom Filters"
3. Skills can now use `{{ value | your_filter }}`

### Adding a Tier 2 plugin
1. Create `plugins/your_plugin.py` with:
   ```python
   TRIGGER = {"type": "command", "command": "yourcommand"}
   # or: TRIGGER = {"type": "cron", "cron": "0 9 * * *", "timezone": "Asia/Jerusalem"}

   async def execute(context):
       # context.http = httpx client
       # context.send_message(text) = send to user
       # context.memory = parsed memory.md content
       return "Message to send"
   ```
2. Test: `/run yourcommand`
3. Commit and push
